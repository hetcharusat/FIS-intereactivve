1.1 Introduction to Information Security

Information Security (InfoSec) is the protection of information systems and networks from unauthorized access, use, disclosure, disruption, modification, or destruction. At the university level, exams focus not just on "what" these are, but "how" they interact structurally.

Core Terminologies

TermDefinitive Academic Meaning
AssetOrganizational resource under evaluation. Divided into Tangible (servers, facilities) and Intangible (IP, reputation, raw data).
ThreatAn external/internal circumstance or event with the potential to adversely impact organizational operations.
VulnerabilityA weakness in an information system, system security procedures, internal controls, or implementation that could be exploited by a threat source.
ExploitThe specific mechanism or technique used to take advantage of a vulnerability.
RiskA measure of the extent to which an entity is threatened by a potential circumstance or event. Typically expressed as a function of the adverse impacts (Impact) and the likelihood of occurrence. Risk = Threat Likelihood × Vulnerability × Impact.

The CIA Triad

The foundational bedrock of InfoSec (referencing Forouzan).

  • Confidentiality: Guaranteeing that data is not disclosed to unauthorized individuals, processes, or devices. Enforced via Encryption, Access Control, and Steganography. Defends against network sniffing and traffic analysis.
  • Integrity: Guarding against improper information modification or destruction. Ensures Non-Repudiation and Authenticity. Enforced via cryptographic hash functions (SHA), digital signatures, and strict role segregation.
  • Availability: Ensuring timely and reliable access to and use of information. Sustained through hardware redundancy (RAID), failover clusters, and DoS mitigation infrastructures limit the risk of downtime.
Dr. Khushi Patel often looks for the interdependencies. E.g., Explain how implementing strong confidentiality (heavy TLS encryption overhead) can detrimentally impact availability (due to processor bottlenecking on boundary firewalls). Link these concepts dynamically in long-answer questions.

1.2 Implementation Issues of Security Goals

Theoretical security is absolute (an air-gapped system locked in a vault). Practical security is a constant compromise.

Strategic Challenges and Trade-offs

1. The Usability-Security Inverse Relationship

A highly secure system often creates immense user friction. If passwords must be 20 characters and rotate every 15 days, users write them down (defeating the control). This is known as "Security Fatigue." Exam answer key: Highlight that security must scale with the risk appetite, not overwhelm it.

2. Overhead and Performance Degradation

Cryptographic operations require significant compute overhead. Encrypting data-at-rest requires localized processing power, whereas encrypting data-in-transit (IPsec/TLS) requires robust network packet processing. Both reduce raw performance metrics.

3. Cost-Benefit Analysis (Quantitative vs Qualitative Risk)

Implementing security is a cost center. Organizations struggle to measure the ROI because a successful security program results in nothing happening.
Rule: The Annualized Cost of the Safeguard (ACS) must be less than the Annualized Loss Expectancy (ALE) without the safeguard.

4. Legal, Compliance, and Privacy Overlap

Security mechanisms themselves can breach policy. Inspecting SSL/TLS traffic (to detect malware) involves decrypting employee communications, which may violate regional privacy acts (e.g., GDPR, local wiretapping laws).

1.3 Control Mechanisms for Information Security

Controls are the countermeasures. For an end-semester exam, you must distinguish between Categories/Classes and Types/Functions.

Categories of Controls (The "How")

  • Management / Administrative: The "paperwork" layer. Defines the scope. Examples: Risk assessments, Acceptable Use Policies (AUP), employee onboarding, disaster recovery planning.
  • Technical / Logical: The "digital" layer. Software/hardware enforcing the rules. Examples: Intrusion Prevention Systems (IPS), Next-Gen Firewalls, IAM systems, Multi-Factor Auth (MFA).
  • Physical / Operational: The "real world" layer. Examples: Biometric turnstiles, HVAC systems (for server availability), mantraps, rack locks.

Functions of Controls (The "When")

FunctionPurpose & TimingExample
PreventiveProactive. Stops an incident before it occurs.Firewall dropping unauthorized ingress packets; Security Guard.
DetectiveReactive. Identifies an incident is underway/occurred.IDS alerting a SOC; SIEM log analysis; CCTV monitoring.
CorrectivePost-incident. Reverses the damage to a known standard.Restoring from secure offsite backups; Antivirus terminating a process.
DeterrentPsychological. Discourages an attacker."Warning: Prosecuted to full extent of law" banners; visible cameras.
CompensatingAlternative. Used when primary control is impossible.Adding heavy logical tracking because physical limits prevent a mantrap.

1.4 Access Control - Administrative and Technical

Access control dictates how a Subject (Client/User) interacts with an Object (File/Server). It requires 3 phases: Identification (claiming identity), Authentication (proving identity), and Authorization (evaluating permissions).

Formal Access Control Models

1. Discretionary Access Control (DAC)

The owner (creator) of the object dictates access. Access Control Lists (ACLs) store these permissions. Vulnerability: Susceptible to Trojan horses; a user can easily grant access to an unauthorized party. Used in standard Windows NTFS / Linux Ext4 file systems.

2. Mandatory Access Control (MAC)

The operating system overrides the user. Dependent on strict security labels (Clearances for Subjects, Labels for Objects - e.g., "Top Secret"). It enforces the Bell-LaPadula model (No Read Up, No Write Down). Crucial for military-grade data segregation.

3. Role-Based Access Control (RBAC)

Access decisions are based on the user's role/job function within an enterprise, vastly simplifying administration. User -> Role -> Permission. This is the industry standard for scalable organizational security.

4. Attribute-Based Access Control (ABAC)

The most granular model. Evaluates boolean logic using Subject attributes (Title=Manager), Object attributes (Data=Financial), and Environment attributes (Time=09:00-17:00, Location=OnPremise).

Administrative Principles to master:
- Separation of Duties (SoD): No single transaction can be completed by one subject end-to-end to prevent fraud.
- Principle of Least Privilege (PoLP): Subjects are granted bare-minimum permissions necessary for their role.

1.5 to 1.6 Passwords, Hash Functions, & Threats

Knowledge-based authentication is statistically the weakest link due to physiological limits on memory (low entropy passwords used by humans) and computational advancements in cracking.

Understanding Cryptographic Hashes

Passwords must never be stored in plaintext. They are passed through a deterministic one-way cryptographic algorithm (like SHA-256 or bcrypt) that outputs a fixed-length string (a Digest). When authenticating, the system hashes the provided input and performs a string-comparison against the database.

Core Hash Properties required for InfoSec:

  • Pre-image Resistance: Computationally infeasible to reverse the hash back to the original password.
  • Collision Resistance: Infeasible to find two different input messages that hash to the exact same output.

Threat Vectors & Defenses

Attack TypeMechanismDefense
Brute-ForceAttempting every mathematical permutation of a character space.Account lockout thresholds; requiring longer password lengths (increases permutation space exponentially).
Dictionary AttackUsing extensive pre-compiled lists of common words/passwords from previous breaches.Complexity requirements (Symbols/Numbers); Blacklisting known-bad passwords.
Rainbow TablesMassive databases of pre-computed hash-to-password mappings to instantly reverse a hash.Salting: Appending random bits to a password *before* hashing. Ruins pre-compiled tables.
If the exam asks for advanced password defense, mention Key Stretching algorithms (bcrypt, PBKDF2). These intentionally slow down the hashing process linearly, making brute-force mathematically impossible within a standard human lifetime.

1.7 Multifactor Authentication (MFA) – Enterprise Challenges

True MFA requires combining authentication parameters from at least two different categories.

The Categories (Factors)

  1. Something you know (Knowledge): Passwords, PINs, Security Questions.
  2. Something you have (Possession): Smart Cards, Hardware FIDO/YubiKeys, Authenticator Apps (TOTP), SMS (Deprecated by NIST).
  3. Something you are (Inherence): Fingerprints, Retina scans, Facial geometry.

Implementation & Threat Challenges

MFA is not a silver bullet. You must articulate its vulnerabilities for full marks:

  • SIM Swapping (Possession Bypass): Attackers use social engineering against telecom providers to port the victim's phone number to an attacker-controlled SIM, intercepting SMS MFA tokens.
  • Biometric Inmutability (Inherence Vulnerability): Unlike a password, if someone steals your fingerprint data from a breached database, you cannot "reset" your fingerprint.
  • MFA Fatigue (Push Bombing): Attackers repeatedly trigger login requests until an exhausted employee clicks "Accept" on their phone to stop the notifications.
  • Crossover Error Rates (CER): In biometric systems, the challenge of balancing False Acceptance Rates (FAR - bad guy gets in) against False Rejection Rates (FRR - good guy gets blocked). The equilibrium point is the CER.

1.8 Spheres of Information Security

Information security is visualized as concentric spheres, representing the architectural concept of Defense-in-Depth (Layered Security). If an attacker breaches one layer, they are immediately confronted with a different type of control mechanism.

Management / Policy Layer (Outer Sphere)
AUPs, ISMS Frameworks, Legal Compliance
Network & Host Systems (Middle Sphere)
Firewalls, IDS/IPS, Antivirus, Access Control Lists
Core Information Asset
Data-at-Rest Encryption

Exam Context: The core of the sphere represents the data (the ultimate target). The outer spheres are the security layers. Discuss the NSTISSC Security Model (The McCumber Cube) which models 3 dimensions: Security Goals (CIA), Data States (Storage, Processing, Transmission), and Countermeasures (Technology, Policy/Practice, Human factors). The intersection of these cubes represents complete spherical coverage.
